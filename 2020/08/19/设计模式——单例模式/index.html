一、单例模式

  所谓单例，就是整个程序有且仅有一个实例。该类负责创建自己的对象，同时确保只有一个对象被创建。

  单例模式（Singleton Pattern）是 Java 中最简单的设计模式之一。这种类型的设计模式属于创建型模式，它提供了一种创建对象的最佳方式。

  这种模式涉及到一个单一的类，该类负责创建自己的对象，同时确保只有单个对象被创建。这个类提供了一种访问其唯一的对象的方式，可以直接访问，不需要实例化该类的对象。比如Hibernate的sessionFactory,它充当数据存储源的代理，并负责创建session对象，显然这并不是轻量级的对象，通常情况下，一般只需要一个sessionFactory就够了，此时就应用了单例模式了

1、作用：

  保证一个类仅有一个实例，并提供一个访问它的全局访问点。

2、解决的问题：

  一个全局使用的类频繁地创建与销毁。

3、何时使用：

  当您想控制实例数目，节省系统资源的时候。

4、如何使用：

  判断系统是否已经有这个单例，如果有则返回，如果没有则创建。

二、单例模式的实现

  • 饿汉式
    • 静态常量
    • 静态代码块
  • 懒汉式
    • 传统懒汉式
    • 同步方法
    • 同步代码块（省略）
  • 双重检查
  • 静态内部类
  • 枚举

    1、写法一（饿汉式——静态常量）

    a、步骤

    Ⅰ、构造器私有化

Ⅱ、类的内部创建对象

Ⅲ、对外提供一个静态的公共方法（getInstance）

b、代码实现

//饿汉模式——静态常量
class StaticFinalSingleton{
    //构造器私有化
    private StaticFinalSingleton() {}
    //类的内部创建对象
    private final static StaticFinalSingleton instance = new StaticFinalSingleton();
    //对外提供一个静态的公共方法
    public static StaticFinalSingleton getInstance() {
        return instance;
    }
}

c、测试代码

@Test
public void testStaticFinalSingleton() {
    System.out.println("饿汉式");
    StaticFinalSingleton singleton01 = StaticFinalSingleton.getInstance();
    StaticFinalSingleton singleton02 = StaticFinalSingleton.getInstance();

    System.out.println(singleton01 == singleton02);

    System.out.println("singleton01.hashcode=" + singleton01.hashCode());
    System.out.println("singleton02.hashcode=" + singleton02.hashCode());
}

d、运行结果

"饿汉式"
true
singleton01.hashcode=1555093762
singleton02.hashcode=1555093762

e、分析

Ⅰ、优点：

  写法简单，在类装载的时候完成实例化，避免了线程同步

Ⅱ、缺点：

  没有达到lazy loading的效果，如果从始至终没有使用到这个类的话，就造成了内存浪费

Ⅲ、总结：

  这种方式基于classloder机制避免了多线程同步的问题，不过，instance在装载类时就实例化，在单例模式中大多都是调用getInstance方法，但是导致类装载的原因有很多种，因此不确定有没有其他的方式，或者其他的静态方法导致类装载，这时候初始化instance就没有达到lazy loading效果，所以虽然这种单例模式可用，但也有可能造成内存浪费

2、写法二（饿汉式——静态代码块）

a、步骤

Ⅰ、构造器私有化

Ⅱ、类的内部声明单例对象

Ⅲ、在代码块中、创建单例对象

Ⅳ、对外提供一个静态的公共方法（getInstance）

b、代码实现

//饿汉模式——静态代码块
class StaticCodeSingleton{
    //构造器私有化
    private StaticCodeSingleton() {}
    //类的内部声明单例对象
    private static StaticCodeSingleton instance;
    //在代码块中、创建单例对象
    static{
        instance = new StaticCodeSingleton();
    }
    //对外提供一个静态的公共方法（getInstance）
    public static StaticCodeSingleton getInstance() {
        return instance;
    }
}

c、测试代码

@Test
    public void testStaticCodeSingleton() {
        System.out.println("饿汉式");
        StaticCodeSingleton singleton01 = StaticCodeSingleton.getInstance();
        StaticCodeSingleton singleton02 = StaticCodeSingleton.getInstance();

        System.out.println(singleton01 == singleton02);

        System.out.println("singleton01.hashcode=" + singleton01.hashCode());
        System.out.println("singleton02.hashcode=" + singleton02.hashCode());
    }

d、运行结果

饿汉式
true
singleton01.hashcode=1555093862
singleton02.hashcode=1555093862

e、分析

  这种方式和上面的静态常量其实是类似的，只不过实例化的方式放在了静态代码块中，也是在类装载的时候，就执行静态代码块的中的代码，所以它的优缺点其实和静态常量是差不多的，所以这种方式可用，但是也会造成内存浪费

3、写法三（懒汉式——传统懒汉式）

a、步骤

Ⅰ、构造器私有化

Ⅱ、类的内部声明单例对象

Ⅲ、提供一个静态的公有方法

Ⅳ、调用时判断是否已存在此类对象

b、代码实现

//懒汉式——传统懒汉式
class LazySingleton{
    //构造器私有化
    private LazySingleton() {}
    //类的内部声明单例对象
    private static LazySingleton instance;
    //提供一个静态的公有方法
    public static LazySingleton getInstance() {
        //调用时判断是否已存在此类对象
        if (instance == null) {
            instance = new LazySingleton();
        }
        return instance;
    }
}

c、测试代码

@Test
public void testLazySingleton() {
    System.out.println("懒汉式");
    LazySingleton singleton01 = LazySingleton.getInstance();
    LazySingleton singleton02 = LazySingleton.getInstance();

    System.out.println(singleton01 == singleton02);

    System.out.println("singleton01.hashcode=" + singleton01.hashCode());
    System.out.println("singleton02.hashcode=" + singleton02.hashCode());
}

d、运行结果

懒汉式
true
singleton01.hashcode=1555093762
singleton02.hashcode=1555093762

e、分析

Ⅰ、优点

  确实起到了lazy loading效用

Ⅱ、缺点

  但只能在单线程的情况下使用，即线程不安全

Ⅲ、结论

  虽然实现了lazy loading，但是对于项目来说，这种方式可用性不高

4、写法四（懒汉式——同步方法）

a、步骤

Ⅰ、构造器私有化

Ⅱ、类的内部声明单例对象

Ⅲ、提供一个静态的公有方法同时给他上锁

Ⅳ、调用时判断是否已存在此类对象

b、代码实现

class LazySynchronizedMethodSingleton{
    //构造器私有化
    private LazySynchronizedMethodSingleton() {}
    //类的内部声明单例对象
    private static LazySynchronizedMethodSingleton instance;
    //提供一个静态的公有方法同时给他上锁
    public static synchronized LazySynchronizedMethodSingleton getInstance() {
        //调用时判断是否已存在此类对象
        if (instance == null) {
            instance = new LazySynchronizedMethodSingleton();
        }
        return instance;
    }
}

c、测试代码

@Test
public void testLazySynchronizedMethodSingleton() throws InterruptedException {
    System.out.println("懒汉式");
    LazySynchronizedMethodSingleton singleton01 = LazySynchronizedMethodSingleton.getInstance();
    LazySynchronizedMethodSingleton singleton02 = LazySynchronizedMethodSingleton.getInstance();

    System.out.println(singleton01 == singleton02);

    System.out.println("singleton01.hashcode=" + singleton01.hashCode());
    System.out.println("singleton02.hashcode=" + singleton02.hashCode());
}

d、运行结果

懒汉式
true
singleton01.hashcode=1555093762
singleton02.hashcode=1555093762

e、分析

Ⅰ、优点

  起到了lazy loading效用,同时解决了线程不安全的问题

Ⅱ、缺点

  但是因为加了synchronized锁，所以每一次执行getInstance方法时都需要进行同步处理

Ⅲ、结论

  在实际开发种并不推荐这种方式

5、写法五（双重检查）

a、步骤

Ⅰ、构造器私有化

Ⅱ、类的内部使用volatile声明单例对象

Ⅲ、对外提供一个静态的公共方法（getInstance）

Ⅳ、在方法中判断单例对象是否为空

Ⅴ、使用synchronized同步代码块

Ⅵ、代码块中继续判断单例对象是否为空

b、代码实现

class DuplicationCheckSingleton{
    //构造器私有化
    private DuplicationCheckSingleton() {}
    //类的内部使用volatile声明单例对象
    private static volatile DuplicationCheckSingleton instance;
    //对外提供一个静态的公共方法（getInstance）
    public static DuplicationCheckSingleton getInstance() {
        //在方法中判断单例对象是否为空，如果不等于空直接返回对象
        if (instance == null) {
            //使用synchronized同步代码块。假如多个线程进入则再判断一次
            synchronized (DuplicationCheckSingleton.class){
                //代码块中继续判断单例对象是否为空
                if (instance == null) {
                    instance = new DuplicationCheckSingleton();
                }
            }
        }
        return instance;
    }
}

c、测试代码

@Test
    public void testDuplicationCheckSingleton() throws InterruptedException {
        System.out.println("双重检查");
        DuplicationCheckSingleton singleton01 = DuplicationCheckSingleton.getInstance();
        DuplicationCheckSingleton singleton02 = DuplicationCheckSingleton.getInstance();

        System.out.println(singleton01 == singleton02);

        System.out.println("singleton01.hashcode=" + singleton01.hashCode());
        System.out.println("singleton02.hashcode=" + singleton02.hashCode());
    }

d、运行结果

双重检查
true
singleton01.hashcode=1555093762
singleton02.hashcode=1555093762

e、分析

总结

  起到了lazy loading效用,同时解决了线程不安全的问题,也提高了效率，Double-Check的概念是多线程开发中经常用到的。

  

6、写法六（静态内部类）

a、步骤

Ⅰ、构造器私有化

Ⅱ、创建一个静态内部类

Ⅲ、静态内部类定义一个实例化类属性

Ⅳ、使用getInstance返回静态内部类中实例化的类

b、代码实现

class StaticInnerClassSingleton{
    //构造器私有化
    private StaticInnerClassSingleton() {}
    //创建一个静态内部类
    private static class SingletonInstance{
        //静态内部类定义一个单例的静态类属性，并且实例化该类
        private static final StaticInnerClassSingleton INSTANCE =
                new StaticInnerClassSingleton();
    }
    //使用getInstance返回静态内部类中实例化的类
    public static StaticInnerClassSingleton getInstance() {
        return SingletonInstance.INSTANCE;
    }
}

c、测试代码

@Test
public void testStaticInnerClassSingleton() {
System.out.println("静态内部类");
StaticInnerClassSingleton singleton01 = StaticInnerClassSingleton.getInstance();
StaticInnerClassSingleton singleton02 = StaticInnerClassSingleton.getInstance();

System.out.println(singleton01 == singleton02);

System.out.println("singleton01.hashcode=" + singleton01.hashCode());
System.out.println("singleton02.hashcode=" + singleton02.hashCode());
}

d、运行结果

静态内部类
true
singleton01.hashcode=1555093762
singleton02.hashcode=1555093762

e、分析

Ⅰ、这种方式采用了类装载的机制来保证初始化实例只有一个线程
Ⅱ、静态内部类在StaticInnerClassSingleton装载时并不会立即实例化，而是需要实例化时才调用getInstance方法，从而装载SingletonInstance类，从而实现StaticInnerClassSingleton类的实例化
Ⅲ、类的静态属性只会在第一次加载类时初始化，所以在这里，其实时JVM在保证线程的安全性，因为在类的初始化时，别的线程是无法进入的
Ⅳ、这种方式实现了懒加载，解决了线程安全性、效率高，所以推荐使用

三、单例模式的优缺点

优点：

  1. 在内存里只有一个实例，减少了内存的开销，尤其是频繁的创建和销毁实例（比如管理学院首页页面缓存）。
  2. 避免对资源的多重占用（比如写文件操作）。

    缺点：

  3. 没有接口，不能继承，与单一职责原则冲突，一个类应该只关心内部逻辑，而不关心外面怎么样来实例化。

四、使用场景和应用实例

1、使用场景：

  1. 要求生产唯一序列号。
  2. WEB 中的计数器，不用每次刷新都在数据库里加一次，用单例先缓存起来。
  3. 创建的一个对象需要消耗的资源过多，比如 I/O 与数据库的连接等。
  4. 需要频繁创建和销毁的对象，创建对象耗时过多或消耗资源过多（重量级对象）

    2、应用实例

  5. Windows 是多进程多线程的，在操作一个文件的时候，就不可避免地出现多个进程或线程同时操作一个文件的现象，所以所有文件的处理必须通过唯一的实例来进行。
  6. 一些设备管理器常常设计为单例模式，比如一个电脑有两台打印机，在输出的时候就要处理不能两台打印机打印同一个文件。

七大原则——接口隔离原则

2020-08-14- 2020-08-24
32k- 1:05-
设计模式 设计模式七大原则

一、什么是接口隔离原则

定义

  客户端不应该依赖它不需要的接口（Clients should not be forced to depend upon interfaces that they don’t use）。

  类间的依赖关系应该建立在最小的接口上（The dependency of one class to another one should depend on the smallest possible interface）。

释意

  接口隔离原则是对接口的使用进行约束规范的一个原则，它告诉我们要想把接口用好，关键在于隔离。隔离，指断绝接触、断绝往来。那么我们使用接口时，要隔离什么东西呢？对于上述定义的第1点，“客户端不应该依赖它不需要的接口”，这里的隔离是指客户端和它不需要的接口隔离，也就是客户端不要使用它不需要的接口，这个很容易理解，在实践中也很容易实现。我们着重看一下第2点，“类间的依赖关系应该建立在最小的接口上”，它要求“最小的接口”，也就是该接口中没有多余的方法，所以这里的隔离是指和多余的方法隔离。

  综上所述，接口隔离原则告诉我们，不要把一大堆方法塞进一个接口里，导致这个接口变得臃肿无比。应该要根据实际需要，让接口中只有用得上的方法，也就是说要细化我们的接口。

二、为什么需要接口隔离原则

  建立单一接口，不要建立庞大臃肿的接口，尽量细化接口，接口中的方法尽量少。为类提供专属的接口定制服务。

  下文例子中，将一个庞大的接口变更为 3 个专用的接口所采用的 就是接口隔离原则。在程序设计中，依赖几个专用的接口要比依赖一 个综合的接口更灵活。接口是设计时对外部设定的“契约”，通过分散定义多个接口，可以预防外来变更的扩散，提高系统的灵活性和可维护性。
  说到这里，很多人会觉的接口隔离原则跟之前的单一职责原则很相似，其实不然。其一，单一职责原则原注重的是职责；而接口隔离原则注重对接口依赖的隔离。其二，单一职责原则主要是约束类，其次才是接口和方法，它针对的是程序中的实现和细节；而接口隔离原则主要约束接口接口，主要针对抽象，针对程序整体框架的构建。接口的设计粒度越小，系统越灵活，这是不争的事实。

三、关于接口隔离原则的例子

1.需求

  定义一个或多个接口，需要有吃、跑、说、飞、游五个方法,定义一个people类和一个动物类，并实现接口中对应的方法，定义一个学生，完成对人类这个类的吃、跑、说方法，定义一个天鹅，完成对动物类中的吃、飞、游方法

2.方案一

public interface Interface{
    String eat();
    String run();
    String say();
    String fly();
    String swim();
}
public class People implements Interface{

    @Override
    public String eat() {
        return "在吃饭";
    }

    @Override
    public String run() {
        return "在跑步";
    }

    @Override
    public String say() {
        return "在说话";
    }

    @Override
    public String fly() {
        return "在坐飞机";
    }

    @Override
    public String swim() {
        return "在游泳";
    }
}
public class Student {
    public void eat(Interface anInterface) {
        System.out.println("学生" + anInterface.eat());
    }

    public void run(Interface anInterface) {
        System.out.println("学生" + anInterface.run());
    }

    public void say(Interface anInterface) {
        System.out.println("学生" + anInterface.say());
    }
}
public class Animal implements Interface{

    @Override
    public String eat() {
        return "在觅食";
    }

    @Override
    public String run() {
        return "在行走";
    }

    @Override
    public String say() {
        return "在叫";
    }

    @Override
    public String fly() {
        return "在飞翔";
    }

    @Override
    public String swim() {
        return "在游泳";
    }
}
public class Swan {
    public void eat(Interface anInterface) {
        System.out.println("天鹅" + anInterface.eat());
    }
    public void fly(Interface anInterface) {
        System.out.println("天鹅" + anInterface.fly());
    }
    public void swim(Interface anInterface) {
        System.out.println("天鹅" + anInterface.swim());
    }
}
package com.blog.principle.interfaceIsolation;

public class InterfaceIsolation01 {
    public static void main(String[] args) {
        Student student = new Student();
        People people = new People();
        student.say(people);
        student.eat(people);
        student.run(people);

        Swan swan = new Swan();
        Animal animal = new Animal();
        swan.eat(animal);
        swan.fly(animal);
        swan.swim(animal);
    }
}

运行结果：

学生在说话
学生在吃饭
学生在跑步
天鹅在觅食
天鹅在飞翔
天鹅在游泳

3.方案二

interface Interface02_1{
    String eat();
}
interface Interface02_2{
    String run();
    String say();
}
interface Interface02_3{
    String fly();
    String swim();
}
class People02 implements Interface02_1,Interface02_2{

    @Override
    public String eat() {
        return "在吃饭";
    }

    @Override

        return "在跑步";
    }

    @Override
    public String say() {
        return "在说话";
    }
}
class Student02 {
    public void eat(Interface02_1 anInterface) {
        System.out.println("学生" + anInterface.eat());
    }
    public void run(Interface02_2 anInterface) {
        System.out.println("学生" + anInterface.run());
    }
    public void say(Interface02_2 anInterface) {
        System.out.println("学生" + anInterface.say());
    }

}
class Animal02 implements Interface02_1, Interface02_3{
    @Override
    public String eat() {
        return "在觅食";
    }

    @Override
    public String fly() {
        return "在飞翔";
    }

    @Override
    public String swim() {
        return "在游泳";

}
class Swan02 {
    public void eat(Interface02_1 anInterface) {
        System.out.println("天鹅" + anInterface.eat());
    }
    public void fly(Interface02_3 anInterface) {
        System.out.println("天鹅" + anInterface.fly());
    }
    public void swim(Interface02_3 anInterface) {
        System.out.println("天鹅" + anInterface.swim());

package com.blog.principle.interfaceIsolation;

public class InterfaceIsolation02 {
    public static void main(String[] args) {
        Student02 student02 = new Student02();
        People02 people02 = new People02();
        student02.say(people02);

四.注意

  1. 接口尽量小，但是要有限度。对接口进行细化可以提高程序设计灵活性是不挣的事实，但是如果过小，则会造成接口数量过多， 使设计复杂化。所以一定要适度。

  2. 为依赖接口的类定制服务，只暴露给调用的类它需要的方法， 它不需要的方法则隐藏起来。只有专注地为一个模块提供定制服务， 才能建立最小的依赖关系。

  3. 提高内聚，减少对外交互。使接口用最少的方法去完成最多的事情。

  4. 运用接口隔离原则，一定要适度，接口设计的过大或过小都不好。设计接口的时候，只有多花些时间去思考和筹划，才能准确地实践这一原则。

五.接口隔离VS单一职责原则

  接口隔离推荐接口尽量细化，同时接口中的方法尽量少。这里看起来和单一职责原则类似。那么它们有什么区别呢?

  接口隔离和单一职责的审视的角度是不相同的

  单一职责的要求是类和接口职责单一，注重的是职责。是业务逻辑的划分，

  接口隔离的原则需要接口的方法尽量少。通过“尽量使用多个专门的接口”

  总的来说前者针对单个类的,后者针对类之间的依赖耦合关系
